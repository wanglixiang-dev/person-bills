# 个人记账本规格与技术方案评审结果

## 1. 评审范围

本次评审基于当前仓库中可用文档：

- `PRD.md`
- `SPEC.md`

## 2. 总体结论

当前 PRD 和 SPEC 已覆盖个人记账 MVP 的核心闭环：注册登录、分类、账单 CRUD、月度汇总、分类饼图和数据隔离。整体方向可行，适合作为轻量级 MVP 起点。

但在进入后端开发前，仍存在几类关键缺口：

- 账号体系规则未锁定，验证码注册、密码设置、未注册邮箱自动创建账号之间存在产品和安全冲突。
- 消费预警在 PRD 中是核心价值之一，但 SPEC 将其暂缓，仅预留预算表，范围不一致。
- 分类编辑/删除、账单删除撤销、未来日期、注销、多币种等仍是待确认项，可能影响数据库和接口设计。
- 安全设计还不够完整，缺少验证码防刷、JWT 过期/刷新、CSRF/XSS、审计日志、敏感数据保护策略。
- SQLite 可用于 MVP，但多用户并发、部署备份、迁移到生产数据库的策略需要提前明确。

建议：MVP 可以继续推进，但必须先锁定“账号注册登录策略、消费预警是否纳入首期、分类删除策略、验证码安全策略、SQLite 部署边界”这 5 个决策。

## 4. 分维度评审

### 4.1 需求完整性

已覆盖内容：

- 邮箱验证码注册/登录、邮箱密码登录、退出登录。
- 账单新增、查看、编辑、删除。
- 默认分类和自定义分类。
- 月度收入、支出、结余统计。
- 分类支出占比饼图。
- 多用户数据隔离。

建议补充的功能点：

- 预算与消费预警的完整闭环：预算设置、预算修改、阈值、提醒入口、超额状态、关闭提醒。
- 账单筛选：按月份、分类、收入/支出类型筛选。
- 账单搜索：按备注或金额范围搜索，可放 P1。
- 数据导出：CSV/Excel 导出，财务产品建议至少规划 P1。
- 数据备份与恢复：SQLite 文件备份、用户数据导出、迁移策略。
- 密码管理：设置密码、修改密码、忘记密码、仅验证码账号后续补设密码。
- 账号注销流程：注销确认、数据删除范围、冷静期或立即删除。
- 分类管理完整规则：编辑、删除、排序、图标/颜色、自定义分类归属。
- 空状态和新手引导：首次登录后如何引导用户完成第一笔账单。
- 报表扩展：月度趋势、收入支出趋势、分类明细排名。

优先级建议：

- P0：账号规则、账单 CRUD、分类新增、月度统计、数据隔离。
- P1：预算预警、筛选、导出、分类编辑/删除。
- P2：多币种、趋势分析、分类图标颜色、复杂搜索。

### 4.2 边界条件

当前文档已有部分边界：邮箱格式、验证码位数、金额大于 0、无账单月份、收入不计入支出饼图、数据隔离。

仍需补充的异常情况：

- 验证码：重复发送、过期、错误次数超限、同邮箱/同 IP 频控、验证码被重复使用。
- 账号：邮箱大小写归一化、邮箱别名处理、用户已存在但未设置密码、密码弱口令。
- 金额：最大金额、小数位截断/四舍五入、前端输入 `0.001`、负数、科学计数法、超大整数。
- 日期：未来日期、闰年 2 月 29 日、月底跨月、服务端时区与用户时区不一致。
- 分类：重复分类、分类名空格、超长分类名、删除被账单引用的分类。
- 账单：编辑时切换收支类型但分类不匹配、删除后统计刷新失败、并发编辑同一账单。
- 报表：支出总额为 0 时比例除零、比例四舍五入后合计不等于 100%、分类被删除后的历史展示。
- 网络：保存中断、重复提交、弱网重试、接口超时、离线状态。
- 分页：账单数据量过大时列表分页或按月懒加载。

建议在 SPEC 中补充“边界条件矩阵”，每个边界至少包含前端处理、后端校验、错误码和验收用例。

### 4.3 技术可行性

整体技术栈 FastAPI + Vue3 + SQLite 对 MVP 可行，学习成本低、开发速度快。

主要实现难点：

- SQLite 并发写入能力有限：多用户同时新增/编辑账单时，需要合理设置连接池、事务和 busy timeout。
- 验证码系统不是简单接口：需要邮件服务、频控、过期、重试、日志和安全防刷。
- 报表统计需要稳定口径：后端应负责统计，避免前端和后端口径不一致。
- 分类删除会影响历史账单：若硬删除分类，历史账单展示会丢失分类名；建议软删除或在账单中保留分类快照。
- JWT 退出登录语义较弱：如果只前端删除 token，已签发 token 在过期前仍有效。
- 后续数据库迁移：如果未来迁到 PostgreSQL/MySQL，需避免 SQLite 特有 SQL 和过度依赖本地文件。

技术建议：

- 后端采用 SQLAlchemy + Repository/Service 分层，隔离数据库实现。
- SQLite 开启外键约束，定义唯一约束和索引。
- 金额统一用整数分存储，前端输入元、后端二次校验。
- 报表接口由后端聚合，前端不自行计算核心统计。
- 邮件验证码用独立服务类，开发环境 mock，生产环境配置真实 provider。
- 对账单新增/编辑接口增加幂等保护或前端防重复提交。

### 4.4 安全风险

当前已有的安全设计：

- 密码 bcrypt 哈希。
- 验证码哈希存储。
- JWT 鉴权。
- 不信任前端传入的 `user_id`。
- CORS 白名单。
- SQLite 文件不提交代码仓库。

仍需补强的安全项：

- JWT 有效期：必须定义 access token 过期时间，建议短期 token。
- Refresh token：如需要长期登录，必须设计刷新令牌、轮换和失效策略。
- 验证码防刷：邮箱、IP、设备维度限流；验证码错误次数限制。
- 登录风控：连续失败后临时锁定或增加等待时间。
- 敏感日志：禁止记录明文验证码、密码、完整 token。
- XSS 防护：分类名、备注等用户输入必须长度限制并安全渲染。
- CSRF：如果未来改为 Cookie 存 token，需要 CSRF token 或 SameSite 策略。
- 数据越权：所有资源查询、编辑、删除都必须使用 `id + user_id`。
- 账号枚举：注册、登录、验证码发送接口的错误提示需避免暴露账号是否存在。
- 数据备份安全：SQLite 文件备份需要权限控制和加密存储策略。

高优先级安全决策：

1. Token 存储位置：localStorage、sessionStorage 还是 HttpOnly Cookie。
2. Token TTL 和刷新策略。
3. 验证码发送和校验限流策略。
4. 登录失败策略。
5. 生产环境 HTTPS 和 CORS 配置。

### 4.5 用户体验

当前体验方向合理：移动端优先、首页即主界面、快速记账、月度概览。

建议改进：

- 新手首次进入后展示“记第一笔”快捷引导。
- 新增账单默认选择“支出 + 当天 + 最近使用分类”，减少操作步数。
- 常用分类排序应按使用频率动态调整，而不是固定顺序。
- 删除账单提供撤销，比二次确认更流畅。
- 编辑入口不要只依赖长按，增加显式编辑按钮或点击进入编辑。
- 月份切换可支持左右滑动和按钮双入口，兼顾可发现性。
- 饼图旁展示分类排行榜，避免用户只看颜色难以理解。
- 对“收入/支出/结余”增加颜色和语义区分，但避免过度依赖颜色。
- 账单列表建议按日期分组，长列表更容易浏览。
- 保存失败时保留用户输入，不要清空表单。

## 5. 待确认问题

以下问题应在开发前确认并回写 PRD/SPEC：

1. 注册是否必须设置密码？如果不设置，后续如何补设密码？
2. 验证码登录时未注册邮箱是否自动创建账号？
3. 邮箱验证码发送是否接入真实邮件服务？服务商、频控和失败策略是什么？
4. Token 有效期、刷新机制和退出登录失效策略是什么？
5. 消费预警是否纳入首期？如果纳入，预算维度和提醒方式是什么？
6. 分类是否支持编辑/删除？删除被历史账单引用的分类如何处理？
7. 删除账单使用二次确认、撤销，还是软删除回收站？
8. 是否允许未来日期账单？
9. 默认人民币是否长期固定？多币种是否进入后续路线图？
10. SQLite 是仅用于本地 MVP，还是计划用于线上多用户服务？
11. 是否需要账单导出或数据备份能力？
12. 月度统计按哪个时区计算？
13. 账单列表是否需要分页、筛选和搜索？

## 7. 评审结论

当前文档达到 MVP 方向定义和初步技术设计水平，但还没有达到“可无歧义开发”的程度。

建议结论：有条件通过。

进入正式开发前必须完成：

- 锁定账号注册/登录策略。
- 锁定消费预警首期范围。
- 锁定分类删除和账单删除策略。
- 补齐验证码、JWT、越权防护等安全细则。
- 明确 SQLite 部署边界和备份迁移策略。
- 补齐缺失的 `checklist.md` 和 `tasks.md`，建立需求到任务再到验收的追踪关系。
